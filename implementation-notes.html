Key decisions

1. Full redraw now clears native scrollback — but only on a true resize

Why the override. Iris previously refused to clear native scrollback on any full redraw and instead repainted only the visible viewport slice with \x1b[J. In practice that left stale/duplicated rows after width/height resizes (the reported scrollback/redraw bug). pi-mono clears screen+home+scrollback (\x1b[2J\x1b[H\x1b[3J) and rebuilds from state, which is the robust fix.

The split (important). “Full redraw” is not one thing. Two distinct paths now exist in terminal_surface.rs:

  • Resize redraw (write_resize_redraw): triggered only by a width change, or a height change when not Termux. Emits \x1b[2J\x1b[H\x1b[3J and rewrites all lines from index 0, so the history scrolls back into the freshly-cleared scrollback instead of being lost or duplicated. Writing all lines is required precisely because 3J wipes the native scrollback; only the app can repopulate it. Transcript state is already bounded, so this is acceptable.
  • In-place rebuild (existing write_full(clear=true), used for non-append shrink / change-above-viewport within write_diff_or_replay): keeps the viewport-down \x1b[J clear and repaints the visible slice. This path must not clear scrollback (e.g. closing the slash palette must not wipe the user's history view).

Preserved invariants. First render writes everything with no clear; append/diff updates never clear scrollback; only genuine resize wipes+rebuilds it.

2. Termux exception

Termux changes terminal height every time the soft keyboard shows/hides. A scrollback-clearing redraw on every toggle would replay the whole history and churn scrollback. So when TERMUX_VERSION is set, a height-only change (width unchanged) skips the resize-redraw path and falls through to the normal diff/replay (scrollback preserved). Width changes still force a full resize redraw, even under Termux (wrapping genuinely changed). Detection is read once at construction into a termux field (test-overridable) to avoid process-global env races in parallel tests.

Oracle-flagged correctness fix. A height-only change reaching the diff path must recompute the effective previous viewport top for the new height before any movement / “above viewport” math, or a shrink could patch rows that are no longer visible and corrupt the viewport. write_diff_or_replay now does this (mirrors pi-mono's prevViewportTop = heightChanged ? max(0, prevBufferLen - height) : prevViewportTop).

3. 16ms render coalescing (scheduler design)

An active turn emits bursts of agent events; drawing on each one is wasteful and flickers. RenderScheduler in tui_loop.rs coalesces requests to roughly one draw per 16ms: the first request after a draw (or after idle) draws immediately; further requests within the window collapse into one pending flag flushed at the interval boundary by a sleep_until select branch. When nothing is pending the branch is disabled, so the loop stays CPU-idle (no timer wakeups).

  • Scope. Wired into run_turn (the bursty path). Idle/modal draws stay immediate — they are one-per-keystroke, not bursts, and immediate echo is better UX. Startup and turn-boundary draws bypass the scheduler (forced immediate).
  • Testability. The scheduler is pure and clock-injectable (poll(now) -> DrawNow | Wait(deadline) | Idle), unit-tested with injected Instants — no real-time sleeps. The 16ms latency added to coalesced input is imperceptible; the first event after idle still draws immediately.

4. Changed-line rendering

The existing changed-range diff path is preserved: single-line updates rewrite only the changed visible rows (\x1b[2K), and unsafe changes (shrink, change above the viewport) fall back to the in-place rebuild. Deletion/shrink stale-row clearing keeps using the viewport-down clear so scrollback is not corrupted (per the split in §1).

5. Over-width diagnostics — fail fast, no silent clipping

Previously render_line silently clipped any over-width line to the terminal width, hiding renderer wrap bugs (and still risking corruption at wide-char/ANSI boundaries). Now render_lines measures each line's visible display width; if a line exceeds the terminal width it writes a diagnostic log and returns an error before any corrupting write. Higher-level layout (screen.rs) is responsible for wrapping to width, so an over-width line is a real bug the guard surfaces.

Crash-log path (Iris data convention, not .pi). The log is written to $IRIS_CRASH_LOG_DIR if set, else ~/.iris/tui-crash.log, matching the ~/.iris convention used by settings/auth/sessions. Logging is best-effort and never masks the original error. The directory is a test-overridable field, so the over-width test asserts both the error and the log without touching the real home dir.

6. IME hardware-cursor markers

Contract: a focused composer/editor emits a zero-width marker (CURSOR_MARKER, an APC sequence) at its cursor cell; the terminal surface finds it, records the cursor (row, col), strips it before any terminal write, and positions the hardware cursor for IME candidate-window placement.

  • Marker as a structured span, not embedded ANSI. Rather than embedding the APC string in the rendered ANSI and re-parsing it with ANSI-aware width (pi-mono's approach), Iris carries the marker as a ratatui Span whose content equals CURSOR_MARKER. render_line detects that span, records the column as the accumulated display width of preceding spans, and skips emitting it. This keeps width accounting and the diff source of truth free of escape noise. The marker never reaches previous_lines.
  • Producer. screen.rs locates the editor's reversed block cursor (ratatui-textarea draws it with Modifier::REVERSED) within the editor text area and injects the marker span just before that cell — only when the composer owns focus (!turn && no modal && no approval).
  • Positioning runs on every render, including Unchanged (oracle fix): when only the cursor moves within an otherwise-identical row, the stripped strings do not change, so cursor positioning is independent of the changed-range diff.
  • Show vs position. Default is “position but keep hidden” (the hardware cursor is moved for IME but stays hidden because Iris draws its own reversed block cursor, avoiding a double cursor) — matching pi-mono's opt-in showHardwareCursor. A test-only flag exercises the show path. Hidden when no marker; restored (shown) on shutdown via restore().

7. Keyboard protocol negotiation/restoration

Setup previously pushed only DISAMBIGUATE_ESCAPE_CODES unconditionally. Now Iris probes supports_keyboard_enhancement() first and, when supported, pushes DISAMBIGUATE_ESCAPE_CODES | REPORT_EVENT_TYPES | REPORT_ALTERNATE_KEYS (flag set 7, matching pi-mono). Requesting event types is safe because every Iris key handler gates on Press/Repeat and ignores releases. A keyboard_enhanced flag records whether the push happened, so the matching pop runs exactly once on shutdown/error and never leaks a stray pop on terminals that never negotiated. Negotiation is best-effort: a probe error or push failure does not abort startup.

Deliberate non-parity. pi-mono's modifyOtherKeys (CSI >4;2m) fallback is not emitted: Crossterm does not model or parse it, and Crossterm still delivers usable key events without the protocol. The Crossterm-supported fallback (simply not pushing when unsupported) is used instead. The enable/disable sequences are factored into pure helpers and unit-tested with a fake writer.

Deliberate deviations from pi-mono (summary)

  • Cursor marker travels as a structured span, not an embedded APC literal re-parsed from ANSI.
  • No modifyOtherKeys raw fallback (Crossterm has no model for it).
  • No KITTY image handling / over-width image exemptions (Iris renders no images), so the over-width guard has no image exemption branch.
  • Coalescing is wired only into the active-turn burst path; idle/modal draws remain immediate.
  • Crash log lives under ~/.iris (Iris data convention), not ~/.pi/agent.

Tests

  • terminal_surface.rs: first render no-clear; width resize and non-Termux height resize clear scrollback (2J/H/3J) and rebuild the full document; Termux height-only no scrollback clear; Termux width still clears; append/diff no full clear; changed-line-only diff; shrink stale-row clearing; over-width errors + writes crash log (no clipping); exact-width not flagged; cursor marker stripped + hardware cursor positioned (incl. on Unchanged); show-cursor opt-in.
  • screen.rs: focused composer emits the marker; a running turn does not.
  • tui.rs: keyboard-enhancement pushed only when supported; popped only when enabled (lifecycle/restoration) via a fake writer.
  • tui_loop.rs: render coalescing state machine (immediate first draw, burst coalesced then flushed at the 16ms boundary, idle after draw) with injected clocks.

Known limitations

  • The over-width guard uses unicode-width measurements, which can differ from a specific terminal's glyph widths for some emoji/CJK; this matches the prior clipping path's measurement and is the standard tradeoff.
  • IME candidate-window placement relies on the terminal honoring a moved-but-hidden cursor position; the opt-in show path exists for terminals/users that prefer a visible hardware cursor.

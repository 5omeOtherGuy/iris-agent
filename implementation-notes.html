ADR-0026 — assembler

  • Deleted from system_prompt/mod.rs: Source/source_rank (fragments are now sourceless), load_repo_fragments, repo_fragments_dir, contain_in_workspace, repo_provides_fragments, ensure_default_fragments, materialize_defaults, default_file_contents, global_fragments_dir, load_dir, parse_fragment, strip_frontmatter_open, split_at_closing_fence, dedup_by_name, MAX_FRAGMENT_BYTES. Frontmatter parsing went with file loading: internal fragments are structured data, not text to parse.
  • Kept: build_prompt (pure core), take_anchor (simplified to "last occurrence wins"), order_middles (slot then name; source tie-break removed), push_block, generated tool blocks, discover_project_docs/read_doc_in_dir/read_regular_bounded/open_no_follow (the AGENTS.md path-safety reader), date helpers.
  • assemble_defaults kept as the single #[cfg(test)] seam (used by provider request-shaping tests); it is not a second runtime path — assemble is the only runtime entry and differs only by doing real project-doc discovery.
  • defaults.rs: dropped the description field (it existed only for materialized frontmatter; keeping it would be dead code under -D warnings) and the template fragment + TEMPLATE const (a copy-ready template for a file format that no longer exists; it was slot-0-disabled, so the rendered prompt is unchanged).
  • cli.rs: removed ModelSwitch::{system_prompt, set_system_prompt} — their only caller was the old /trust prompt-reassembly path.
  • "No writes to ~/.iris/fragments at startup" is guaranteed structurally (the materialization function no longer exists; the module does no writes at all) plus tests that assemble creates no workspace files and ignores a temp HOME/.iris/fragments/identity.md without materializing or rewriting HOME fragments. The HOME test is serialized under an env lock.

ADR-0027 — store shape

  • File shape: same file (~/.iris/trust.json, IRIS_TRUST_PATH override), still a flat JSON object keyed by canonical (symlink-resolved) cwd. IRIS_TRUST_PATH must be an absolute path outside the project directory; relative or project-local overrides fail closed so a repo-committed file cannot become the grant source. The value changes from the legacy string ("trusted"/"untrusted") to a record: 
    { "/abs/canonical/cwd": {
    "allow_tools": ["write", "edit"],
    "allow_bash": ["cargo test"],
    "allow_bash_prefix": ["git "],
    "sandbox": "restricted"        // optional; stored, not enforced
} }
    Legacy string entries fail the record parse and read as the empty policy (fail closed, pinned by test); they are overwritten on the next grant. No version field: the flat map plus fail-closed per-entry parsing covers the one migration that exists.
  • An emptied record (all grants revoked, no sandbox) removes the project's entry entirely instead of storing an empty object.
  • serde derive (already a workspace dep with derive) is used for the record; no new dependencies.

ADR-0027 — enforcement wiring

  • Tier split: nexus::ProjectPolicy (data: tool set, bash exact set, bash prefix set) is the enforcement-layer shape; wayland::trust::ProjectPolicyRecord is the persistence shape. Nexus never touches disk: the host (main.rs) loads the policy via wayland::trust::policy_for(cwd).to_policy() and injects it with Agent::with_project_policy(policy, sink). New grants persist through the injected ProjectPolicySink trait (wayland::trust::PolicyStoreSink). The /trust//permissions editor now calls wayland::Harness::{project_policy_record,apply_project_policy_edit}, so Tier 3 no longer reads/writes the store directly; enforcement stays in Nexus and IO/edit application stays in Wayland (ADR-0005 direction preserved).
  • Precedence computation: in run_gated_single, auto_approved = !destructive && (session_allowed || project_allowed). All three layers are allow-only (there are no deny grants), so "most-specific wins" degenerates to a union of the allow layers over the prompting global default; the destructive floor is applied before any layer.
  • bash is never grantable per-tool: ProjectPolicy::allows consults the tool-name set only for non-bash tools; bash goes exclusively through exact/prefix command matching, so a blanket bash allow is not expressible in the policy.
  • Prefix matching is token-boundary: git/git  matches git status but not gitevil (bash_prefix_matches), so a stored prefix cannot widen to lexically adjacent commands.
  • [p] offer and defense in depth: ApprovalGate::review gained an allow_project flag, passed as !destructive. Non-obvious invariant placement: even if a front-end returns AllowProject for a destructive call anyway, Nexus refuses to apply or persist the grant (runs the call once, notices) — invariant 2 is enforced at the decision handler, not just at the offer.
  • Destructive classifier hardening after review: bash_command_is_destructive now checks the normalized basename of command tokens: path-qualified commands (/bin/rm, /usr/bin/dd), quoted/escaped command words ('rm', \rm, r\m), and mkfs.* variants (mkfs.ext4). Nexus tests pin that a project prefix grant such as git still re-prompts for git status; /bin/rm -rf sub.
  • A bash call with a missing/empty command cannot mint a grant (allowed once with a notice).
  • A failed sink persist degrades to session-lifetime scope with an honest notice; headless --print loads the policy (grants auto-approve headless too) but installs no sink (its gate cannot return AllowProject).
  • reset_session (/new, in-session /resume) clears session_allowed but keeps project_policy — it is keyed to the cwd, not the conversation.
  • /trust//permissions editor (Harness::apply_project_policy_edit) fails closed: if the store write fails, the in-memory policy is not updated either, so the live session never runs looser than what is persisted. Edits refresh the live agent via agent.set_project_policy; no provider rebuild is needed anymore (the prompt no longer depends on trust).

Deferrals

  • Sandbox-posture enforcement: the sandbox field is stored, round-tripped, shown read-only in the /trust modal, and covered by the invariant-3 test (grant writes never create or alter it). Wiring it into src/tools/bash/sandbox.rs is deferred — it touches the Landlock layer and would balloon the diff; nothing currently writes the field, so there is no way to loosen it.
  • Full /settings review-and-revoke surface: deferred; the /trust//permissions modal covers list/grant(write/edit)/revoke.
  • Text/non-TTY /trust editor: deferred (TUI-only, matching the prior precedent); the text UI approval prompt does offer [p], so grants can still be minted in the text fallback.

Drift found vs. the task prompt

  • The ADR files (and docs/adr/README.md rows) were already merged to origin/main, so remote ADR verification was possible; the "uncommitted local worktree" caveat in the prompt was stale.
  • The prompt claimed bash has supports_allow_always=true; the code (and a pinned test) opts all three mutating tools (bash/write/edit) out of session allow-always. The project layer is independent of that flag, so the design is unaffected — it makes ADR-0027 the only persistence path for all three, which is exactly #209's point.
  • The prompt referenced clear_session_policy (~L645); the actual mechanism is Agent::reset_session clearing session_allowed.

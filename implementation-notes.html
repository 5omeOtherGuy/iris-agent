  • New surfaces read the task record + git-status snapshot + (for detail) the session log for display only. No enforcement/recovery path reads body/sessions or the session log.

Session-bar task badge (Unit 2) — a spec-conflict resolution

Conflict: DESIGN-LANGUAGE §9.1 authoritatively fixes the session-bar state cluster as ±N ◇M (user-dirty count, Iris-unsettled ledger count) and declares itself the winning document on any conflict. Unit 2 asks for a more explicit badge carrying a short task id.

Decision: the badge is an additive fuller tier above the authoritative cluster, not a replacement. A new git drop-level 0 renders ±N ◇M task <id8>; it degrades to the exact spec form ±N ◇M (level 1) before the spec's own drop order (± only → no counts → base) applies. The badge is held while the right-hand context readout collapses (meter → /cap → used), so the id shows only "where width allows" and narrow widths never overflow. Rationale (endorsed by design review): replacing ◇M with ◇<id> would break the Iris-half's "ledger file count" meaning; keeping count-only would fail Unit 2. ◇ always leads the badge, preserving the unsettled-task signal even when the ledger count is zero.

Consequence: git levels renumbered 0..4 (was 0..3); one extra candidate row in session_bar; [WT] now renders at level <= 3 (was <= 2) to keep its old drop width after the shift. Unmerged ▲N still overrides the badge (tested).

Unified task modal (Units 3–4)

  • Folded the active-task view into the existing TaskPicker rather than adding a parallel modal: a non-selectable header (id, body, attribution counts, linked-session count, age) above the selectable recoverable/legacy rows. Enter still emits only AdoptTask; the active task is never adoptable (guarded by TaskCard::is_adoptable filtering the selector rows).
  • Linked-session detail is an in-modal sub-view. Entering it (right arrow) emits ViewTaskSessions(id); the loop fetches the bounded, cwd-scoped extraction by reusing cli::sessions_for_task_lines (the same deterministic path as /sessions) and re-opens the modal with the detail attached. Leaving the detail (left/esc) is pure state (clears the field) — no disk, no harness — so the modal stays disk-free during interaction. Trade-off: re-entering the list after detail resets the selector cursor to the top (the rebuild is fresh). Acceptable for v1.
  • Modal decoupled from git status for rebuild. The active card's identity/body/sessions come from the harness (active_task()); the file counts + age are enriched from the git-status snapshot only when its task id matches the active task (active_card), so a stale snapshot never mislabels a different task's numbers (tested). When unknown, those meta parts are simply omitted.
  • Right-arrow target: the selected recoverable row, else the active task. So an active-only modal can still open its sessions; but when both an active task and recoverable rows exist, the right arrow targets the selected recoverable row. To view the active task's sessions in that case, use /sessions (defaults to the active task) — preserved shortcut.

Projection (Unit 1)

  • One projection type ui::task_view::TaskCard represents both active and recoverable/legacy tasks; the old row-only TaskRow was removed in favor of it (row rendering derives from the card).
  • New git-safety DTO ActiveTaskDisplay { task_id, body, sessions } (mirrors AdoptedTask) + GitSafety::active_task_display() + Harness::active_task() expose the live task's opaque payload without leaking the private Task. UI imports the Wayland DTO (inward dependency); Wayland never imports ui::TaskCard.
  • user_files/iris_files are Option: populated for the active card (from the matched git-status split), left None on recovery rows (the attribution split is a git-status concept not recomputed per row). Matches Unit 1's "where available".
  • Added RecoverableTask::is_legacy() (display-only bool) so the UI can mark legacy rows without exposing the private TaskClass enum; recovery policy still uses TaskClass directly.

Deferred / out of scope (unchanged)

  • Model-generated task titles, summarization, model-facing history tools, cross-project search, global index, query DSL, worktree-apply mechanics.
  • No settlement UI in the modal: the active header only points at /git · /diff · /accept · /rollback; those keep routing through existing Harness/GitSafety methods (Unit 6).

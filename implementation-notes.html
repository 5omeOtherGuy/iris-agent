Release-ops validation notes

Decisions and discoveries not already explicit in the task spec. Tracking issue #252; refs PR #233 / issue #199. Concise, factual.

Environment

  • Network is reachable here (api.github.com returned HTTP 200), unlike the spec's "appeared offline" note. This let me install the pinned dist CLI.
  • dist (cargo-dist) was not on PATH; installed cargo install cargo-dist --locked --version 0.28.0 (3m34s compile) → dist 0.28.0. So the tool-driven path (unit 2/3) was taken, not the offline fallback.
  • No git tags, gh release list empty, Cargo.toml version 0.1.0 — re-confirmed; no release ever cut.
  • Host target: x86_64-unknown-linux-gnu. aarch64 / darwin targets are not locally buildable here (no cross toolchains); deferred to CI matrix / operator.

Unit 2 — release.yml: keep hand-written, do NOT regenerate

  • Discovery: dist plan refused to run — the committed release.yml (114 lines, hand-written) diverges from dist generate's canonical output (~291 lines).
  • The canonical output would regress three deliberate choices:
    1. installs dist via curl|sh instead of the pinned cargo install cargo-dist --locked --version (the committed header cites supply-chain posture: crates.io checksum + --locked);
    2. drops IRIS_DIST: "1" on the build job — that env sets the iris_dist cfg marker that update_strategy() requires, so regenerating would silently break self-replace;
    3. pins actions/checkout@v4 + runs-on: ubuntu-20.04, whereas the whole repo standardizes on checkout@v7 + ubuntu-latest (ubuntu-20.04 is retired).
  • Decision: keep the hand-written workflow; add cargo-dist's sanctioned escape hatch allow-dirty = ["ci"] to [workspace.metadata.dist]. Verified: with it, dist plan runs cleanly (no --allow-dirty flag) and dist generate --check refuses to clobber the file. Oracle concurred (8/10; main residual risk = future dist upgrades won't force a regen diff, mitigated by the version-sync guard + a runbook note).
  • Naming invariant confirmed from cargo-dist itself: dist plan emits iris-agent-<target>.tar.gz + .tar.gz.sha256 and installer iris-agent-installer.sh — exactly what selfupdate.rs::asset_name/checksum_name and install.sh expect.

Test seams (needed to exercise real code paths without a public release)

  • install.sh: added IRIS_RELEASE_BASE_URL override for the artifact base directory (default unchanged = the GitHub releases download URL). Lets the installer hit a local HTTP server over the real download → sha256-verify → extract → atomic-install path. Cleaner than /etc/hosts (no root, no TLS/SNI).
  • selfupdate.rs: added a loopback-only IRIS_UPDATE_RELEASES_API_URL override for the releases API endpoint. Guard rejects any non-loopback host, so the env var cannot redirect a real user's update to an attacker server. The real reqwest download, sha256_matches, tar extract, and self_replace still run unchanged. This is a narrow test seam, not a rewrite of the update logic.

Unit 3 — local artifact defect: missing [profile.dist]

  • Defect found & fixed: dist build --artifacts=local failed with "profile `dist` is not defined". The hand-written Cargo.toml never had dist init run against it, so the cargo-dist build profile was missing — the real CI release would have failed too. Added the canonical [profile.dist] (inherits = "release", lto = "thin"). Rebuild succeeded; produced iris-agent-x86_64-unknown-linux-gnu.tar.gz (5.4 MB) + .sha256; recomputed SHA-256 matches the sidecar; archive contains iris + CHANGELOG/README/LICENSE.
  • Discovery: iris has no --version flag (only help/--help/-h exit 0). The spec/README assume iris --version; the smoke test uses iris --help instead. Adding --version is a reasonable follow-up but out of scope here.

Unit 4 — install.sh defect: global-variable clobber

  • Defect found & fixed: POSIX sh functions have no local scope, so verify_checksum() assigned to a bare archive (= the full workdir path), clobbering main()'s archive. The later tar -xzf "$workdir/$archive" then got a doubled path (/tmp/x//tmp/x/iris-agent-...tar.gz) and every real install failed. PR #233 only ran shellcheck (clean) and never executed the installer — exactly the gap this validation closes. Fixed by giving the function underscore-prefixed names (_vc_archive etc.).
  • Validated against the real unit-3 artifact over a local HTTP server (IRIS_RELEASE_BASE_URL override): happy path installs + iris --help exits 0; a corrupted .sha256 hard-fails with no install; the off-PATH note is emitted.

Unit 5 — iris update self-replace (real code path)

  • Used the real dist-built binary (v0.1.0, IRIS_DIST=1 + self-update) as the running exe, a local mock of releases/latest (IRIS_UPDATE_RELEASES_API_URL, loopback-only), and a replacement archive whose iris is the same binary + a trailing sentinel (ELF ignores it) so a successful swap is observable by a changed SHA-256. The real reqwest download, sha256_matches over real bytes, tar extract, and self_replace all run.
  • Proven: (A) newer tag → downloads, verifies, self-replaces (hash changes S→S'); (B) tag == running version → "Already on the latest", binary untouched; (C) corrupted .sha256 → "refusing to install", binary unchanged. Also: the binary takes the SelfReplace strategy (not the cargo fallback), and the loopback guard rejects https://evil.example.com.
  • Source-build fallback (update_strategy() == CargoInstall when the marker/feature are off) is unchanged — covered by the existing select_strategy_* unit tests, untouched.

Deferrals (operator-gated / externally visible / environment-limited)

  • Public v* tag + GitHub release — externally visible, irreversible; runbook step.
  • CARGO_REGISTRY_TOKEN + crates.io publish — repo-admin secret; runbook step.
  • aarch64 / macOS artifacts — no cross toolchains locally; built by the CI matrix. Only the x86_64-linux host target was built here. Naming for all four targets is confirmed from dist plan.
  • Live curl | sh on a clean OS image — no clean image reachable in-session; runbook step.

Review hardening

  • Independent review flagged that the loopback override guarded only the metadata request, not the asset download URLs from the response. Tightened: when IRIS_UPDATE_RELEASES_API_URL is active, the archive and checksum URLs are also loopback-checked before download, so the override can't pull artifacts from a remote host. The default (unset) path still trusts GitHub's real download URLs.
  • install.sh's IRIS_RELEASE_BASE_URL fetches archive and checksum from the same base, so it is a trust override by nature; documented as such (validation, not untrusted mirrors) rather than loopback-restricted, since setting it is equivalent to editing the script the user already runs.
  • scripts/validate-dist.sh now uses a sha256sum/shasum -a 256 fallback so the reproducer runs on macOS (a supported target), matching install.sh's own fallback.

Follow-up risks

  • Future cargo-dist upgrades won't force a regenerated-CI diff (because of allow-dirty); the version-sync test catches the pinned-version half, but the operator must re-diff dist generate when bumping cargo-dist-version. Documented in the runbook.
  • Cross-filesystem replace: self_replace + install.sh both stage beside the target for a same-FS rename; a genuinely cross-FS install dir still relies on self_replace's handling. Not exercised here.
  • iris has no --version; users can't confirm the installed version from the CLI. Suggested follow-up.

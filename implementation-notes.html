2. The redefinition (what changed semantically)

Old model: all footer numbers came from the proposing turn. New (forward) model splits the round:

  • ↓ = output_tokens of the turn that proposed the tool call(s). Set immediately.
  • ↑ = input_tokens - cache_read_input_tokens of the following turn that ingested the tool results.
  • cache = cache_read_input_tokens of that following turn (omit when zero).
  • ctx = input_tokens(following) - input_tokens(proposing) scaled by the context cap (omit without a prior turn or cap).

Key realisation: a single ProviderTurnCompleted(N) event carries both (a) the ↓ for tools proposed in turn N (its output_tokens) and (b) the ↑/cache/ctx for tools proposed in turn N−1 (its input_tokens is the ingestion of N−1's tool results). ctx for the N−1 tools is input(N) - input(N-1), i.e. computed against the existing last_turn_input_tokens baseline before it is advanced.

3. Design choice — delayed-footer patching strategy

The tool footer is rendered during turn N's tool phase, when only ↓ is known; ↑/cache/ctx arrive later at TurnCompleted(N+1), after the footer row is already committed. So already-rendered footers must be patched. Two strategies compared:

StrategyMechanismVerdict
A: row-index map HashMap<call_id, footer_row_index>, adjusted on every splice (active-panel rebuild), truncate (EXPLORE footer pop) and drain (trim_history). Rejected. Fragile: indices shift on splice/pop/trim, and collect_rows temporarily appends rows before splitting them off, so a naive index recorded during a collect is wrong. Every row-moving path would need a fixup.
B: semantic tag + rescan (chosen) Add diag_call: Option<String> to ChromeRow::Footer, tag tool/edit/explore footers with their call id. Patch = update the stored ToolDiag in tool_diags, then rescan self.rows for the tagged footer and rewrite its right string (and the plain text mirror). Rescan is O(rows) but bounded (MAX_TRANSCRIPT_ROWS) and only runs on ProviderTurnCompleted. Chosen. Survives row movement (splice/pop/trim) with no index bookkeeping; fails safely (no-op) if the row was trimmed away. Oracle-reviewed: 8/10, "more robust than index tracking here".

State kept local to Transcript:

  • current_turn_diag: Option<ToolDiag> now holds only ↓ (received) for the current proposing turn.
  • awaiting_input_calls: Vec<String> — call ids proposed by the current proposing turn, awaiting the following turn's input-side numbers. Drained & patched at the next TurnCompleted.

4. Pitfalls found / handled

  • Two event orderings. Normal (TurnCompleted then ToolStarted) adds calls to awaiting_input_calls in assign_turn_diag. Tools-first (a fast tool starts/finishes before its turn completes) adds the still-open/active calls at TurnCompleted when stamping ↓. Both dedup.
  • Do not overwrite ↓. The following turn's patch sets only ↑/cache/ctx and preserves each call's existing received. Turn N+1's own output_tokens becomes the ↓ for N+1's future tools, never the N tools'.
  • ctx baseline order. ctx is computed with the pre-existing last_turn_input_tokens (= input of the proposing turn) before advancing it to the following turn's input.
  • collect_rows. Avoided entirely by not recording row indices (strategy B).
  • Plain text mirror. The patch rewrites both ChromeRow::Footer.right and TranscriptRow.text so /find and collapsed rendering stay consistent.
  • usage: None (Antigravity). A following turn with no usage leaves the footer at ↓-only (awaiting list cleared without a patch); a proposing turn with no usage never sets ↓ so the footer stays clean.
  • tool_diags cap clear. The 1024-entry safety clear could drop a received for a still-awaiting call; documented as an acceptable bound (only under a pathological single session; a dropped entry degrades to no-diag, never wrong numbers).

7. Confirmed facts vs assumptions

  • Confirmed: event ordering (TurnCompleted before tools), cache reads already in input_tokens, single footer construction site, usage: None yields no diag.
  • Assumption: within one round, a following provider turn's input_tokens corresponds to ingesting the immediately-preceding turn's tool results. True for the sequential agent loop; a provider that batches multiple rounds into one request would blur it, but Iris's loop is one provider turn per tool round.
  • Assumption: EXPLORE groups do not span provider turns in practice (a new turn's reasoning/text resets exploring_open via push_blank), so the grouped footer's single tag belongs to the round being patched. Left EXPLORE grouping unchanged (out of scope); if a tool-only turn ever regrouped across a boundary the footer would degrade to the newest round's numbers, never to fabricated ones.
  • Assumption: the 1024-entry tool_diags safety-clear dropping a still-awaiting received is acceptable (only under a pathological single session; degrades to no-diag).

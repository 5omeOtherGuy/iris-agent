Issue 460 implementation notes

  • The primary checkout was dirty and behind origin/main; implementation is in fresh worktree ../iris-subagents-readonly-460 from origin/main.
  • The read-only MVP keeps subagent orchestration in Wayland. Nexus changes are limited to a provider-neutral tool-registry filter so declaration and execution use the same reduced registry.
  • Read-only filtering uses each tool's self-classification: tools are retained only when they are not mutating and do not require approval. This intentionally excludes bash for the MVP even if a specific shell command might be read-only, because the current bash tool classifies itself as workspace-mutating.
  • The MVP handle API stores workers in memory and runs the child on wait. The worker table uses interior mutability so cancel can flip the shared token while wait is awaiting the child. Durable/background recovery is deferred to later subagent slices.
  • Tool allowlists are applied after the read-only capability filter. An allowlist can narrow read-only tools but cannot re-add a mutating or approval-gated tool.
  • Explicit cwd is supported only inside the parent workspace for this read-only slice. Worktree isolation remains rejected until the mutable/worktree issues implement it.
  • max_output_bytes limits the structured result summary returned by this MVP. It is not a provider context-window budget.

Composer / Editor Statusline Upgrade — Implementation Notes

Reviewer-facing notes for the TUI composer statusline upgrade. Records only decisions/changes not explicit in the spec. Routine steps omitted.

Scope & key files

  • src/ui/tui/screen.rs — composer chrome rendering (the file changed by this work).
  • src/ui/tui.rs — constants, style helpers, and the test module for chrome.

Pre-existing state vs. spec

Before this change the runtime status was rendered as a separate line above the editor box (a "rail": ● MODE code ┊ MODEL … ┊ CTX … ┊ CWD … ┊ BRANCH …). The spec explicitly forbids a floating statusline above the composer, so the rail was removed and its data relocated: mode/model/context into the editor's top border, and cwd/git into a quiet unboxed label below the box.

Decisions not specified by the spec

  • Active dot is statically orange. The top-frame ● uses the accent color always (it signals "session active"); it is not tied to turn state. The inline working indicator already signals turn activity.
  • Effort is always shown when set (e.g. LOW), including during an active turn. The removed rail hid effort while the spinner ran; the new frame shows it because the spec layout includes it. It is dropped only by width fallback.
  • Model display = uppercased bare model id (gpt-5.4 → GPT-5.4), not the catalog display name (GPT 5.4), because the spec shows GPT-5.4. Effort is appended uppercased.
  • CTX label = catalog context-window label uppercased (300k → CTX 300K). The meter is computed from total_tokens / parse(window) where parse handles k/m suffixes and plain integers.
  • Meter value persisted separately from per-turn usage. Added Footer.context_used_tokens, updated on every ProviderTurnCompleted and not cleared at turn start, so the meter does not blank at the start of every turn. It is cleared in set_footer_with_context when the model or context window changes, so a prior model's usage is never shown against a new window (oracle-recommended). The working indicator keeps using Footer.usage (still per-turn).
  • Meter math: 10 dots, div_ceil so any nonzero usage lights ≥ 1 dot; over-budget clamps to a full strip. The last lit dot is the orange edge; earlier lit dots are muted (DarkGray); empty dots are dim (○).
  • Width step 3 ("shorten model alias") is a no-op. No alias table exists in the catalog, so narrowing is handled by the remaining steps: drop effort → drop meter → drop CTX → minimum ● CODE ─ MODEL. Candidates are monotonic (effort never reappears after being dropped).
  • Pulsing not implemented. The spec's edge-pulse / 100%-pulse are "may" behaviors; omitted (static styling only).
  • Workspace label placement: indented to the transcript text column (TEXT_COLUMN_X_PADDING = 4), dim, formatted <cwd> ┊ git <branch> with no trailing separator. CWD uses a middle-ellipsis that preserves the final path segment (project name); if even …/<project> does not fit it left-ellipsizes the project name.

Layout / frame-integrity mechanics

  • The statusline is printed into the editor's top border via buf.set_line() after the block, body, and hint are rendered (so the textarea cannot overwrite it at tiny heights). The line is built at exactly the box width: ┌─ + segments joined by ─ + + ─ fill + ┐.
  • Never breaks the frame: if even the minimum content cannot fit (extremely narrow terminals), composer_top_border returns None and the plain Block border is kept. A test sweeps box widths 6..=200 asserting the line is exactly the box width and is corner-bounded.
  • Height budget reordered. Chrome is now [menu, editor, workspace-label] (label below the editor). The editor frame is protected first: the menu yields to MIN_EDITOR_H, then the workspace label yields to both, so constrained heights drop the label before squeezing the editor frame. Constant GLOBAL_STATUS_H renamed WORKSPACE_LABEL_H.

Code-review outcomes (4 parallel reviewers)

Reviewed from four angles: correctness/frame-integrity, spec conformance, architecture/tier boundaries, and test quality. No correctness or spec-conformance bugs were found; the frame-integrity width sweep and meter math were validated. Applied fixes:

  • Removed duplicated helpers (AGENTS.md reuse rule): the local spans_display_width and take_first_display now reuse wrap::spans_width and wrap::truncate_to_width. take_last_display is kept (wrap only exposes a prefix variant) but no longer allocates a String per character.
  • Stale usage reset: set_footer_with_context now also drops the carried-over usage (not just the meter value) when the model/context changes, so the working indicator can't briefly show a prior model's IO. Verified the model-switch test still passes (it keeps the same model).

Considered and deferred / not actioned (with rationale):

  • Extract a chrome.rs module (reviewer: medium). screen.rs is ~1.2k lines, but it was already that size from the concurrent #163 refactor; splitting is a separate cohesion refactor outside this task's scope. Deferred as a follow-up to keep this change focused.
  • Meter uses total_tokens (input+output), not input_tokens. Verified the provider contract: for the stateless Codex/Anthropic lanes input_tokens is the full cumulative prompt, so total_tokens is a sound, slightly-conservative proxy for window fullness. Kept as-is (tested); switching to input_tokens is a possible future refinement.
  • Case-sensitive model comparison in the meter-reset check: all live callers pass the bare lowercase model id, so this cannot misfire today.
  • Ultra-narrow degradation: below the minimum content width the top border returns None and the plain Block border is kept (status hidden, frame intact). This is intentional graceful degradation, covered by the width sweep (no panic / correct corners).

Caveats / follow-up risks

  • Concurrent refactor mid-task: commit #163 ("split TUI rendering modules") landed while this work was in progress, moving chrome rendering from src/ui/tui.rs into src/ui/tui/screen.rs. This change targets the post-refactor layout; tests remain in the tui.rs test module.
  • No explicit light/dark theming added. The spec lists per-mode color tables; the implementation reuses the existing semantic style helpers (border = Color::Gray, accent ORANGE which this codebase maps to Color::Yellow, dim via Modifier::DIM) and relies on the terminal palette, consistent with the rest of the TUI. The exact light/dark hex directions are not encoded.
  • Meter granularity is coarse (10 buckets, ceil). One token of usage shows one lit dot; this is intentional per the spec ("each dot ~10%").
  • The set_footer_with_context model-equality check compares the bare model display string. Both the live loop and the test helper pass the bare model id, so the comparison is consistent; a future caller passing a qualified id would reset the meter more eagerly (safe direction).

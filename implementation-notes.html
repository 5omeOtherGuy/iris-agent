Anthropic Messages OAuth stream robustness — implementation notes

Branch anthropic-stream-robustness. Records decisions not explicit in the spec, tradeoffs, and reviewer context. Not a step log.

Verified failure being fixed

Anthropic stream ended before content_block_stop [status=200 ...]. The parser only emits this when message_stop arrives while open_tools or open_reasoning is non-empty; plain text blocks cannot trigger it.

Key decisions

  • Duplicate-delta prevention (scope item 5) — chosen strategy: gate retry on "no visible text emitted this attempt", NOT per-attempt buffering. Rationale: buffering every attempt would kill live token streaming on the happy path (every turn, not just retried ones). Instead the parser carries an explicit emitted_visible_text flag (set only when a non-empty text_delta is successfully forwarded to the sink). A malformed status-200 stream is retried only when no visible text was emitted on that attempt. Because tool input (input_json_delta) and reasoning (thinking_delta/signature_delta) are buffered internally and never streamed live, the verified tool-only / thinking-only malformed turns always qualify for retry. If visible text was already streamed and then the stream malformed, we do NOT retry; we surface the typed protocol error (DoD permits "retried/recovered OR fails with a typed sanitized protocol error"). A retried attempt by definition emitted no text, so the next attempt starts with no prior visible text and duplicate user-visible deltas are impossible.
    Caveat for reviewers: this predicate is safe only because text_delta is the ONLY live event today. If reasoning/tool deltas ever stream live, the predicate must be revisited.
  • Transport retry placement (scope item 3): added a new shared run_with_retry alongside the existing run_with_reauth rather than generalizing the latter. Antigravity stays on run_with_reauth with identical behavior; only Anthropic uses the retry loop. classify_http_status (Reauth/Fatal) is left untouched for Antigravity; a separate classify_http_status_retryable (Reauth/Retry/Fatal) is added for Anthropic. Attempt::Retry is added to the shared enum; run_with_reauth treats it as fatal for exhaustiveness (Antigravity never produces it).
  • Backoff constants: mirrored from Codex (MAX_TRANSIENT_RETRIES=3, BASE_BACKOFF=500ms, MAX_BACKOFF=8s, up to 250ms jitter, integer Retry-After seconds honored). Intentional small duplication with Codex: Codex predates the shared transport module and keeps its own copy (see transport.rs module note); unifying both is out of scope.
  • Typed protocol error: StreamProtocolAnomaly carries only block-shape metadata (message_stop seen, open tool/reasoning counts, open block indexes, last event type) — never streamed content. It is attached via anyhow::Error::new(..) and the safe request diagnostics are added with .context(..) so the typed error stays downcastable for classification (per oracle review). Network send errors and 429/5xx also become Attempt::Retry; the transient budget is shared across all three and is not reset by the one-shot reauth.
  • max_tokens (scope item 8): base max_tokens changed from a fixed 8192 to the selected model's output_cap (DEFAULT_OUTPUT_CAP=64000 for unknown models). For manual-budget thinking, resolve_manual_thinking(output_cap, budget, output_cap) keeps budget_tokens < max_tokens, so the invariant holds and max_tokens=output_cap. This directly removes the 8192 truncation that produced silently-dropped stop_reason=max_tokens.
  • stop_reason (scope items 6/7): parsed from message_delta.delta.stop_reason, mapped to a provider-neutral CompletionReason enum (EndTurn, ToolUse, MaxOutputTokens, ContextWindowExceeded, StopSequence, Paused, Refusal, Other). Decision change: Other is a unit variant (not Other(String)). Keeping the enum small and Copy avoids tripping clippy's large_enum_variant lint once completion_reason is embedded in AssistantTurn (which flows through ProviderEvent::Completed / StreamResult::Completed); the alternative was boxing AssistantTurn in three enums across six files, a larger blast radius. The raw unknown stop-reason token is still surfaced for diagnostics via a tracing::debug line at parse time (the token is an enumerated wire value, never response content), so observability is preserved without storing it. Truncation reasons (MaxOutputTokens / ContextWindowExceeded) additionally emit a tracing::warn so they are not silently dropped. Stored on AssistantTurn and propagated to AgentEvent::ProviderTurnCompleted.
  • UiEvent left unchanged: completion_reason is added to the Nexus AgentEvent::ProviderTurnCompleted (the provider-neutral metadata event) but NOT to UiEvent::ProviderTurnCompleted, since no UI surface renders it yet. The ui/mod.rs mapping drops it. This keeps the change minimal and leaves typed completion metadata available for later continuation/fallback work, per the goal.

Review follow-up fixes (second pass)

  • Retry-gate invariant hardened (review item 1): the duplicate-delta predicate is unchanged in behavior, but the doc comment on protocol_anomaly_retryable now states an explicit INVARIANT: any future live UI event (live reasoning or tool deltas) MUST set the same emitted_visible_text gate, or the retry policy must change. Added malformed_stream_after_visible_text_is_not_retryable covering the content_block_stop variant after a real text_delta (the prior tests only covered the missing-message_stop variant).
  • Truncation is now user-visible (review item 2): a provider-neutral completion_reason_notice(CompletionReason) lives in nexus.rs and the provider loop emits AgentEvent::Notice right after ProviderTurnCompleted when the reason is MaxOutputTokens or ContextWindowExceeded. Decision: wording is model-neutral ("The model hit its maximum output-token limit..." / "...context-window limit...") rather than the Anthropic-specific "Claude ..." phrasing the request suggested, because the notice is emitted from provider-neutral Nexus completion metadata and the runtime must not hard-code a provider/model name (ADR-0003 / tier boundaries). The existing tracing::warn in the Anthropic adapter is retained. Notices ride the existing AgentEvent::Notice -> UiEvent::Notice path; no new event type.
  • Empty completed turns with a valid stop reason (review item 3): AnthropicStreamParser::finish now treats a terminal stream with no text/reasoning/tool calls as a valid empty completion when a stop_reason was present (completion_reason.is_some()); a contentless stream with NO stop reason still errors as before. The presence of the stop reason (any mapped value, including Other for unknown-but-present tokens) is the gate, since a stop reason is the provider's terminal completion signal.
  • Stale-doc fixes (review item 4): the module headers in transport.rs and anthropic_messages.rs no longer claim "no transient backoff"; they now describe run_with_retry (Anthropic) vs run_with_reauth (Antigravity, reauth-only).

Review follow-up fixes (third pass — 6 reviewer findings)

  • Mid-stream read failure is now retryable when safe (finding 1): a for_each_sse_event failure in send_once previously always returned Attempt::Fatal, even though a pre-stream send() failure was already retried. A mid-stream connection drop/timeout is transient and is now retried on the SAME gate as a protocol anomaly: only when !cancel.is_cancelled() && !emitted_visible_text, so a retry can never duplicate user-visible output and cancellation is never swallowed. No new unit test: there is no HTTP-mock harness for send_once (provider tests operate at the parser level), and the decisive predicate is the already-tested emitted_visible_text gate plus the shared cancellation token; building a mock server purely for this path would dwarf the fix.
  • Defensive redaction of serde errors (finding 2): both serde_json::from_str sites (ingest_event SSE frame, finalize_tool tool JSON) now drop the serde error via map_err(|_| ...) and return a fixed message. Both deserialize to an untyped Value, so serde only produces positional syntax errors (never the input bytes) — this is belt-and-suspenders, not a live leak. Removing the last Result::context call made the anyhow::Context import unused; it was dropped.
  • force_refresh in run_with_reauth (finding 3): investigated the suggested "reset for consistency with retry_loop" and found a reset is actually DEAD CODE here (clippy unused_assignments): this loop's only loop-back is the Reauth arm, which immediately sets force_refresh = true, so a reset to false is never read on any path. Kept the original (no reset) and documented WHY it correctly differs from retry_loop (whose Retry arm loops without touching the flag and thus genuinely needs the reset). No behavior change.
  • Empty-turn test coverage (finding 4): empty_turn_with_valid_stop_reason_completes now also iterates pause_turn and refusal, the other contentless stop reasons finish() accepts.
  • Content-less refusal notice (finding 5): completion_reason_notice now takes had_visible_content and returns "The model declined to respond." for a Refusal that produced no text/reasoning/tool calls. A refusal that DID carry an explanation emits no extra notice (the user already sees it). had_visible_content is captured in the agent loop before reasoning is consumed. Wording stays provider/model-neutral. Tested both ways in nexus_tests.rs.
  • Diagnostics key consistency (finding 6): StreamProtocolAnomaly's Display now uses the last_event= key (was last_event_type=) to match AnthropicDiagnostics and the original failure log, so logs are searchable on one field.

Deferred (explicitly out of the smallest-correct scope)

  • Automatic in-stream continuation / non-streaming fallback (scope item 12): not implemented. Recovering a malformed stream after visible text without a UI rewind event would require buffering all live deltas (regressing streaming UX for every turn) or a new Nexus "discard partial" event. Both exceed the smallest correct fix. Typed error classification, bounded retry/recovery, safe diagnostics, stop-reason propagation, and the max-token cap correction are all implemented (not deferred).

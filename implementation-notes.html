Align arbitrary harness restrictions to pi-mono

Branch feat/pimono-limits off origin/main at 6b48611 (fetch succeeded, primary 0 ahead / 0 behind). Reference tree ~/vendor/pi-mono (consulted, not vendored). Only decisions/discoveries not already explicit in the spec are recorded below, organized by unit.

Verification / drift

  • All <read_first> Iris constants matched the prompt snapshot when re-read (line numbers had drifted a little but the values were as cited).
  • pi-mono path drift: the prompt cited RetrySettings at packages/coding-agent/src/config.ts#L27-L32. The actual type lives in packages/coding-agent/src/core/settings-manager.ts (RetrySettings L27-31: maxRetries default 3, baseDelayMs default 2000; consumed via getRetrySettings() L799-803). The 60s ceiling is maxRetryDelayMs in packages/ai/src/types.ts (L149/L192). All other cited pi-mono files exist as referenced; harness/utils/truncate.ts confirms DEFAULT_MAX_BYTES = 50 * 1024.

Unit 1 — turn cap: unbounded default, optional graceful soft cap

  • Removed the hardcoded MAX_TOOL_ROUNDTRIPS = 50. The loop in Agent::complete_turn is now an unbounded loop with a manual roundtrip counter; it ends naturally when the model emits no tool calls (or on cancellation).
  • Tradeoff / runaway guard: with no fixed cap a malfunctioning model could loop indefinitely. Cancellation (Ctrl-C, the turn CancellationToken) is the always-on runaway guard and is checked at the top of every iteration, before the provider stream, and during tool execution — so a runaway turn is always interruptible. This matches pi-mono's runLoop, which has no maxTurns/maxSteps.
  • Config shape: new Settings.max_tool_roundtrips: Option<usize> (camelCase maxToolRoundtrips). Absent = unbounded. Precedence: project-overridable (merged like contextTokenBudget) because a turn cap can only narrow a run, never redirect tokens. Threaded into the agent via a builder Agent::with_max_tool_roundtrips(Option<usize>), called in run_agent/resume_agent right where context_token_budget() is read.
  • When set and reached, the loop reuses the existing graceful Notice path ("stopped after N tool round-trips; send another message to continue.") + TurnComplete — never a fatal error.

Unit 2 — parallel cap removed

  • Removed MAX_PARALLEL_TOOL_CALLS = 8. run_parallel now uses .buffered(calls.len().max(1)).
  • Ordering approach: kept buffered (not buffer_unordered) deliberately: it preserves result order, which the caller relies on when it zips outcomes back onto the calls. The whole parallelizable run executes concurrently (pi-mono's Promise.all over the batch). max(1) guards the empty-batch case (buffered(0) stalls); in practice run_tools only calls this with a non-empty parallelizable run. is_parallelizable and the sequential/exclusive path are unchanged.

Unit 3 — inline-output threshold to 50 KB

  • MAX_INLINE_TOOL_OUTPUT_BYTES 16 KiB → 50 * 1024. Preview sizes (PREVIEW_HEAD_BYTES = 4K, PREVIEW_TAIL_BYTES = 2K, sum 6K) are unchanged; the compile-time invariant assert!(MAX_INLINE_TOOL_OUTPUT_BYTES > HEAD + TAIL) still holds, so no preview re-sizing was needed. Reuses the existing handle-offload (issue #61) — no new spill subsystem.

Unit 4 — single shared, configurable retry policy

  • New src/mimir/retry.rs with RetryPolicy (max_retries, base_backoff, max_backoff), Copy so cloned providers carry it. It owns the single backoff_delay() (exponential + jitter + bounded Retry-After). Deleted the duplicated constants (MAX_TRANSIENT_RETRIES/BASE_BACKOFF/MAX_BACKOFF) and the two copies of backoff_delay from transport.rs and openai_codex_responses.rs.
  • Defaults adopted (pi-mono-aligned): max_retries = 3, base_backoff = 2000 ms, max_backoff = 60 s. This changes Iris's prior values (was 500 ms base / 8 s cap) — an intentional alignment. Retry-After honoring and jitter (≤250 ms) are preserved; the pathological-hint clamp is now 4 × max_backoff.
  • Config shape: new Settings.retry: Option<RetrySettings> with maxRetries / baseDelayMs / maxDelayMs (all optional; absent subfields fall back to the defaults). Global-only (merged like promptCacheRetention): retry volume affects provider load/cost, so an untrusted project file must not crank it up.
  • Divergence from pi-mono: pi-mono's RetrySettings also has enabled and a per-provider provider override, and keeps the 60 s ceiling in a separate ai/types field (maxRetryDelayMs). Iris keeps one flat policy and exposes the cap as maxDelayMs in the same object (single seam, no inert knobs). No enabled/per-provider knobs were added (they would trigger nothing today).
  • Threading: RetryPolicy is resolved in ModelSelection::resolve from settings.retry_settings() (alongside cache_retention/context_management) and carried on ModelSelection.retry_policy. build_provider passes it to the Codex and Anthropic constructors; a runtime /model switch carries the same policy. Antigravity uses run_with_reauth (reauth-only, no transient retries) and is unchanged.

Unit 5 — bash default timeout removed

  • Removed DEFAULT_BASH_TIMEOUT_SECS = 120. One-shot and session run now derive timeout as input.timeout.filter(|s| *s > 0).map(Duration::from_secs) — unset (or 0) means no caller time limit (pi-mono parity); a positive value is the per-call limit.
  • Kept rail: the persistent-session SESSION_HARD_CAP (600 s) is a wedge-safety rail (a wedged shell, e.g. an unterminated here-doc, still gets killed), not the arbitrary 120 s default. It is unchanged; render_session already reports it when no per-command limit is set.
  • The background-job finalize wait default (previously 120 s when unset) was also removed: unset/0 now blocks until the job completes (cancellation-aware), a positive value bounds the wait.
  • Cancellation, drain, and the temp-file save of full output are untouched. DESCRIPTION/parameter strings updated ("No timeout by default").

Unit 6 — default truncation byte cap to 50 KB

  • text::DEFAULT_MAX_BYTES 1 MB → 50 * 1024. This propagates to read/grep/ls/find/bash display truncation. DESCRIPTION strings and the read paging notice updated "1MB" → "50KB".
  • Decoupling (avoids an unintended shrink): bash/jobs.rs DEFAULT_JOB_CAPACITY previously aliased DEFAULT_MAX_BYTES. It is a peak-memory capture ring bound, not a display cap, so it was pinned to 1_000_000 rather than following the display cap down to 50 KB. The bash one-shot MAX_CAPTURE_BYTES (4 MiB/stream) memory rail is unchanged (only its comment updated to the new 50 KiB display window).

Unit 7 — grep clamps removed

  • Deleted MAX_CONTEXT, MAX_GREP_LIMIT, MAX_HEAD_LIMIT, MAX_OFFSET and all clamping. positive_cap lost its max parameter (now default + reject-zero only). context is no longer clamped; offset has no ceiling; an unset headLimit now shows every output row (was implicitly capped at 500), bounded only by limit.
  • Kept DEFAULT_GREP_LIMIT = 100 and GREP_MAX_LINE_LENGTH = 500. DESCRIPTION/parameter strings updated (dropped "capped at 1000"/"capped at 500").
  • Accepted tradeoff (reviewer-flagged): with context/limit no longer clamped, a request such as context=1000000 makes the match sink collect more context lines before the final truncate_head (2000 lines / 50 KB) trims the output. This is intentional pi-mono parity: pi-mono's grep likewise has no max-context/limit clamp. Memory stays bounded by the actual file content (ripgrep emits no more context than the file has, and overlapping context is merged) and by GREP_MAX_LINE_LENGTH = 500 (the real guard against minified-line blowups). Re-adding a context/limit bound would reintroduce the very clamp this unit removes, so it was deliberately not done.
  • Deferral (named): DEFAULT_CONTEXT stays at 2 (pi-mono defaults to 0). Per the spec, changing the default is a behavior choice beyond removing clamps — divergence noted, not changed.

Unit 8 — ls scan hard cap removed

  • Deleted LS_SCAN_HARD_LIMIT = 20_000 and both its enforcement sites (the limit.min(cap) and the per-directory scan break). The listing is bounded by the requested limit only. Kept DEFAULT_LS_LIMIT = 500 and DEFAULT_FIND_LIMIT = 1000.

Tests changed / added (and why)

  • nexus_tests.rs: tool_loop_stops_gracefully_at_roundtrip_limit → split into tool_loop_stops_gracefully_at_configured_soft_cap (sets a cap via the new builder, proves the graceful Notice + exact provider call count) and new tool_loop_is_unbounded_by_default (scripts 60 rounds past the old ceiling, proves no soft-cap notice). Old test depended on the removed MAX_TOOL_ROUNDTRIPS.
  • nexus_tests.rs: safe_tool_parallelism_is_bounded → safe_tool_parallelism_is_uncapped (batch of 12 > old cap of 8; asserts the whole batch overlaps). Old test asserted the removed cap.
  • transport.rs + openai_codex_responses_tests.rs: retry-loop tests now pass &RetryPolicy::default() and read max_retries from it; the three duplicated backoff_delay_* tests were removed (computation moved to mimir::retry::tests, which adds coverage for the new defaults).
  • grep.rs: context_is_capped → context_is_not_clamped; default_output_is_bounded_and_pageable → explicit_head_limit_is_bounded_and_pageable + new unset_head_limit_is_not_clamped.
  • New config tests (max_tool_roundtrips_*, retry_settings_parse_and_default_and_are_global_only), a selection test (resolve_builds_retry_policy_from_settings_with_defaults), and a bash test (bash_unset_timeout_imposes_no_default_limit).
  • Test-coverage limitation: ls's removed scan hard cap (20,000) is not directly unit-tested — exercising it would require a pathologically large directory. The grep clamp-removal tests cover the "accepts values past the old clamps" requirement; existing ls tests prove no regression.

Named deferrals (out of scope, per spec)

  • Compaction engine tied to context_token_budget (pi-mono compaction.ts): a separate roadmap feature, not an arbitrary restriction. contextTokenBudget left exactly as-is.
  • Memory-safety rails kept: READ_TOOL_MAX_BYTES / WRITE_TOOL_MAX_BYTES (100 MB load/write bounds) and MAX_CAPTURE_BYTES (4 MiB/stream peak capture) are OOM rails, not arbitrary harness ceilings — unchanged. So is the bash session SESSION_HARD_CAP (wedge-safety) and the job ring capacity (peak-memory bound).
  • grep DEFAULT_CONTEXT = 2 kept (pi-mono 0) — see Unit 7.
  • Not adopted from pi-mono: its AgentLoopConfig/host-callback architecture, SDK retry classes, shell-output.ts spill machinery, per-provider/enabled retry knobs, and module structure. Behavior was reimplemented idiomatically in Rust; no new dependencies.

Tier-3 TUI component abstraction

Decisions and discoveries not already explicit in the task spec. Organized by scope unit. Not a step log.

Verification of starting state

  • git fetch succeeded; origin/main at 6b48611 (matches the prompt snapshot). The primary checkout was in detached HEAD at d1d9bce (one commit ahead, owned by another worktree branch), but the main branch == origin/main and worktree-preflight.sh reported PASS. Branched feat/tui-component-abstraction off origin/main.
  • ~/vendor/pi-mono/packages/tui/src/tui.ts present (1495 lines); used as conceptual reference only.
  • Cited symbols/lines had drifted (Iris file tui.rs is the module root; the renderers live in src/ui/tui/{rows,pane,panel, transcript,screen}.rs). Worked from observed code.

Unit 1 — Component trait + cursor marker (src/ui/tui/component.rs)

  • Render unit. Iris's currency is Vec<Line<'static>>, so the trait is render(&self, width: usize) -> Vec<Line<'static>> (pi-mono returns string[]).
  • Trait kept minimal (render only). pi-mono's handleInput and invalidate were intentionally NOT added. Iris routes typed crossterm events through the focus/overlay layer, not a stringly-typed input hook; and no Iris component caches render state, so invalidate would be a no-caller stub — forbidden by the spec's anti-stub rule. Both are deferred until a real caller exists (theming/markdown cache → invalidate).
  • Cursor marker. Adopted pi-mono's APC sentinel (CURSOR_MARKER = "\x1b_pi:c\x07") and a take_cursor_position that locates the marker, computes (row, col) by display width, and strips it across spans. Iris uses NO hardware cursor today (the editor draws its own reversed block cursor via ratatui-textarea), so no shipped component emits the marker. The function is still genuinely wired (called by the composition root, unit 6) and fully unit-tested with a marker-emitting fixture — it is the real seam the deferred hardware-cursor/IME work plugs into, today a no-op strip.

Unit 2 — Container (src/ui/tui/component.rs)

  • Container owns Vec<Box<dyn Component>> and composites in order. A separate borrowed composite() helper exists so hot paths can composite &dyn Component without boxing (the transcript renders up to MAX_TRANSCRIPT_ROWS rows per frame).

Unit 3 — Transcript rows migration (rows.rs, pane.rs, transcript.rs)

  • TranscriptRow::render(width, out) renamed to the inherent render_rows (append-based); the Component impl's render_into override calls it (no per-row Vec) and render wraps it.
  • Transcript::render now selects the visible fold-tagged rows (collapsed panels show the capped preview, expanded panels show the full body) into a Vec<&dyn Component> and calls composite. The in-flight stream became pane::streaming_assistant_rows (returns transient TranscriptRows) so it joins the same borrowed composite instead of rendering directly.
  • Byte-identical: composite's extend equals the former push loop; each row's render_into is unchanged. Proven by the existing width_resize_reflows_transcript_from_state and surface_draw_path_replays_history_from_state tests.

Unit 5 — Overlay + focus (overlay.rs, screen.rs, modal.rs, tui_loop.rs)

  • Docked, not floating (key divergence). pi-mono overlays float and composite over the base document at anchor/margin positions. Iris's modal and slash palette occupy a RESERVED region above the composer (allocated by chrome_heights); the editor shifts down. Modeling them as floating would move where they render and break byte-identical output. So overlays stay docked; the abstraction supplies the two pieces pi-mono couples to overlays: explicit focus and a Component render path. A true floating anchor/margin compositor + overlay handles (hide/show/focus/unfocus) are deferred (no Iris floating UI to migrate; adding it now = abstraction with no caller). Recorded in ADR-0024.
  • Explicit focus. FocusTarget { Editor, Palette, Modal } with precedence Editor < Palette < Modal. Screen::focus() is the single source of truth, derived from existing modal/palette state (no new duplicated flags to keep in sync). It replaced the implicit checks in tui_loop.rs: modal_open() (deleted) → focus() == Modal at the modal-phase gates; palette.is_active() → focus() == Palette for the idle chord-suppression and palette-nav branches. Byte-identical because in idle a modal is never open, so focus()==Palette ⟺ palette.is_active(input).
  • Modal + palette as Components. impl Component for Modal (clamps the usize width back to the u16 the picker renderers use). The palette's STATE stays in slash.rs; overlay::PaletteView is its Component render face, reproducing the former screen::render_palette rows exactly. render_editor_chrome now picks the focused overlay (match screen.focus()), renders it once via Component, uses that length for menu height, and paints it through one overlay::render_menu_lines helper.
  • Byte-identical traps handled. The old code had two inset helpers (render_palette, render_plain_menu_lines) with IDENTICAL inner-rect math; collapsed into one. Modal render width was content_width(area.width), which equals the menu inner width, so rendering at the inner width is unchanged. Menu height (len + 2, MAX_MENU_ROWS cap) preserved by computing it from the single rendered line list. Paragraph still does the clipping (no manual truncation added).

Unit 6 — Composition root (screen.rs)

  • render_document_with_chrome_tail assembles transcript + working block + chrome through a root Container of LinesSection components (mirrors pi-mono TUI extends Container), then calls take_cursor_position to strip any marker before the document reaches TerminalSurface.
  • TerminalSurface seam: untouched. The abstraction only produces the final Vec<Line>; diff/append/replay/resize stay in terminal_surface.rs (existing terminal-surface tests still pass).
  • Performance (post-review): the first pass wrapped the whole document (incl. transcript) in cloning LinesSections, an avoidable O(n) full-document copy per frame. Fixed after wave-1 review: the transcript is MOVED into the document as the scrolling base; only the viewport-bounded tail (working indicator + chrome) is composed through the root Container. Separately, Component::render_into was added so the transcript's borrowed composite over thousands of rows allocates no per-row Vec (matches the pre-abstraction push loop).
  • Cursor-marker scan bounded. take_cursor_position runs on the volatile tail only (the cursor only ever lives in the composer chrome), not the full document.
  • Focus allocation. Screen::focus_for(&input) lets the hot callers (idle key handler, chrome render) reuse their existing editor-text snapshot instead of re-joining the buffer; focus() remains for the non-hot modal-phase gates.

Unit 4 — Tool panels: DEFERRED (named follow-up)

  • After unit 3, tool panels already render through the row-level Component path: every panel is a flat sequence of TranscriptRows carrying ChromeRow chrome, and TranscriptRow: Component.
  • Grouping those flat rows into a standalone Panel component would require reworking the in-place panel machinery (panel_end_from, replace_active_exec_panel / replace_active_tool_panel, the splice-based live-exec/tool finalize, trim-safety, and the collapse filter in Transcript::render). That is high regression risk and not safely byte-identical within this session, so it is deferred. The spec explicitly permits deferring units 3/4/5; units 1–2 plus two real migrations (3 and 5) landed.

Other deferrals (out of scope here)

  • Text fallback UI (src/ui/text.rs): left on its current path. It is a line-printer over stdout (no Screen, no TerminalSurface, no Line composition), so it is not a component-rendered surface; migrating it would add an abstraction with nothing to composite.
  • pi-mono component catalog Iris lacks (image/kitty-iterm graphics, autocomplete, kill-ring, undo-stack, configurable keybindings, theming) and a second terminal-diff engine: out of scope; separate roadmap items.
  • Floating overlay compositor + overlay handles + multiple overlays + hardware cursor: deferred (no current floating UI / no hardware cursor in Iris). The CURSOR_MARKER seam is in place for the cursor part.

Trait shape decision (raised in review)

Reviewers asked whether render_into should be the required method and render the provided one (efficient-by-default). Kept render required + render_into provided/overridable: the only hot component is TranscriptRow (which overrides render_into); Modal, PaletteView, LinesSection, and Container naturally produce owned lines, so requiring render_into would add append-style boilerplate for owned data with no efficiency gain. The current shape achieves zero per-row allocation on the hot path without churning the cold components.

Cursor-marker / wrapping constraint (forward seam)

take_cursor_position runs on already-rendered (post-wrap) lines and uses display_width. Note that wrap.rs::char_width clamps every char to width ≥ 1, so the multi-byte CURSOR_MARKER must NOT pass through wrapping as part of real text or it would be miscounted and force premature wraps. No component emits the marker today, so this is latent; the constraint for the deferred hardware-cursor work is: emitters place the marker in its own span and the composition root strips it before any width/wrap computation (which it already does, bounded to the volatile tail). wrap.rs is intentionally left unchanged to preserve byte-identical wrapping of existing content.

ADR-0006 tension and resolution

ADR-0006 says build Iris-native, sized to Iris, do not import app-scale architecture; Ratatui = primitives; Iris owns the terminal surface. This change stays inside that: the abstraction is small (two files), reuses TerminalSurface unchanged, adds no dependencies, vendors no pi-mono/Codex code, and lives entirely in Tier-3 src/ui/tui/. It amends ADR-0006 only by adding a render-abstraction layer above the surface, and explicitly declines the pi-mono floating-overlay/component-catalog scale. Recorded as ADR-0024.

Reviewer-facing risks

  • Menu-region byte-identical (height math, inner-rect insets, Paragraph clipping) — preserved exactly; covered by existing modal/palette tests + new overlay tests.
  • Focus precedence drift (Editor < Palette < Modal; active palette must still suppress global idle chords) — covered by tui_loop tests including idle_chords_yield_to_an_active_palette.
  • Root container per-frame clone (see Unit 6 tradeoff).
  • Cursor: TerminalSurface untouched; no hardware cursor enabled or moved.

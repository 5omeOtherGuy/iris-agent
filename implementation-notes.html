Unit 1 — the engine module

  • Grapheme crate choice: unicode-segmentation (justified). The Rust stdlib has no grapheme-cluster iterator. I empirically confirmed (throwaway test, since removed) that unicode-width 0.2.2's UnicodeWidthStr::width on a whole grapheme cluster already returns the correct terminal width for ZWJ families, flags, VS16, keycaps, and combining marks (e.g. "\u{1F468}\u{200D}\u{1F469}\u{200D}\u{1F467}" → 2). So no custom emoji/east-asian table is needed; the only missing primitive is cluster segmentation. A thin wrapper over unicode-width alone is insufficient because the bug is that the old per-char wrap/truncate loops split clusters (losing the cluster-aware width and emitting broken half-emoji). unicode-segmentation is the canonical UAX#29 implementation and is already in the build graph transitively via ratatui-core (also pulls unicode-truncate), so adding it as a direct dep downloads/builds no new crate — it is a direct-dep promotion. Rejected alternatives: textwrap/ansi-width (heavier, own width tables, would duplicate unicode-width); hand-rolling UAX#29 segmentation (forbidden by the reuse ladder).
  • Two strip variants, one parser. The old code had two divergent handcoded strippers. The engine has one scanner (CSI ESC [ … final; OSC/DCS/PM/APC/SOS string controls terminated by BEL or ST; plus the 8-bit C1 forms) exposed through two output policies: strip_ansi (removes escape sequences, keeps other chars incl. TAB/CR) and clean_text (also drops remaining control chars). clean_text is byte-identical to the old tui/text.rs::strip_ansi_for_text (the \r-splitting / footer-cleaning path) and is re-exported under that name. strip_ansi matches the intent of the old ui/text.rs::strip_ansi for common content; it differs only on malformed / non-SGR sequences (e.g. bracketed-paste ESC[200~), where the proper CSI parser is more correct than the old "consume until first ASCII letter" loop (which ate the following letter).
  • display_width is raw (does NOT strip ANSI); visible_width does. The historic wrap.rs::display_width was raw UnicodeWidthStr::width and several call sites rely on that as a conservative over-estimate (e.g. the tool-output flood guard wrapped_row_estimate, which intentionally counts escapes as width). Keeping display_width raw preserves that behavior exactly. The new visible_width (strip then measure) is the correct measure for escape-bearing strings and is what fixed the ui/text.rs bug.
  • SGR + OSC 8 carry across wrapped lines (Rust model). AnsiState mirrors pi-mono's AnsiCodeTracker: each boolean SGR attribute plus fg/bg stored as raw code strings (e.g. "38;5;240"), and an Option<Hyperlink>. On each physical row break, active_codes() re-emits the open SGR + reopens the OSC 8 link, and close_codes() closes the link + resets SGR at row end. The OSC 8 terminator (BEL vs ST) is preserved per-link because some terminals only make BEL-terminated links clickable (OAuth URLs use BEL) — reopening with the wrong terminator would break clickability on wrapped rows.

Reserved ansi_aware subsystem (deferred feature, scoped allow)

The string-based ANSI-aware wrap_ansi / truncate_ansi / slice_by_column (and the tracker/parser they need) are built and unit-tested but not wired into any render path. The ratatui span pipeline already carries SGR via Span.style, and emitting clickable hyperlinks is explicitly deferred. They live in a pub(crate) mod ansi_aware with a single scoped #![allow(dead_code)] (rather than a whole-file allow, so genuinely dead production code is still caught). This is the foundation a future markdown/hyperlink renderer builds on; today it only preserves OSC 8 (cross-wrap and cross-truncate tests prove the link is reopened on every physical row).

Per-unit migration parity

  • wrap.rs (unit 2/4): width/truncate/plain-wrap now re-export from the engine; the local display_width/char_width/truncate_*/wrap_to_width bodies are gone. The char-by-char loops (push_wrapped_line) now iterate grapheme clusters via a new push_span_str helper, so for ASCII/CJK output is byte-identical and emoji clusters are no longer split. styled_physical_row stays char-based (it matches a plain wrapped string back to per-char styled cells; cluster-safe wrap output still yields the same chars in order).
  • tui/text.rs (unit 3): strip_ansi_for_text is now textengine::clean_text (byte-identical); the handcoded CSI/string-control consumers were removed. ansi_spans still uses ansi-to-tui for ratatui span parsing (kept, see below) but routes its strip through the engine.
  • ui/text.rs (unit 3) — the documented width-bug fix: visible_width used strip_ansi(input).chars().count(), counting a 2-column CJK/emoji glyph as 1. The frame in write_frame computes right-border padding from visible_width, so non-ASCII diff/title content produced a misaligned (overflowing) right border. Now textengine::visible_width (strip + grapheme display width). To avoid a half-fix (review finding), the entire write_frame path is now display-column-consistent: the fits-check (visible_width), the body hard-wrap (wrap_plain breaks by column width, not cluster count), the title truncation (truncate_plain → truncate_to_width), and the top-border fill (visible_width instead of chars().count()). Behavioral effect: ASCII output is byte-identical (one cluster == one column); CJK/emoji content that previously overran or misaligned the fixed-width frame now wraps and pads to fit. wrap_plain still returns the original ANSI-colored line unchanged when it fits, so colored diff lines keep their color.
  • terminal_surface.rs (unit 5) — the width-math seam: its clip_to_width now calls textengine::clip_to_width and the direct unicode_width import is gone. The old clip was per-char with width unwrap_or(0) (no max(1)); the engine clip is per-grapheme. These are byte-identical for ASCII/CJK/combining (all existing tests are ASCII and stay green). The only behavioral difference is at the exact right edge: an emoji ZWJ/flag cluster that no longer fits is dropped as a unit instead of being split into a broken half-cluster (which the terminal would have rendered as garbage anyway). A new wide_glyph_line_clips_on_cluster_boundary_without_overrun test pins the CJK behavior. The diff/replay/cursor logic was not touched.

ansi-to-tui kept, not replaced

ansi_spans still uses ansi-to-tui to turn raw ANSI into ratatui Spans with proper Style. Replacing it would mean hand-mapping every SGR code to a ratatui Style — out of scope and a regression risk. The task only requires that stripping and width go through the engine, which they now do; ansi-to-tui remains the span-construction path.

wayland/mod.rs NOT migrated (deliberate, architecture-driven)

wayland::truncate_chars (summary MAX_SUMMARY_CHARS/MAX_EXCERPT_CHARS bounds) was left in place and is not routed through the engine. Two reasons: (1) src/wayland is Tier 2 and docs/ARCHITECTURE.md requires Tier 2 to depend on Tier 1 only; routing it to the Tier-3 src/ui engine would create a forbidden upward Tier 2→Tier 3 dependency. (2) Its semantics differ — it is a char-count bound on stored summary text with a "..." ellipsis, not terminal display-width truncation, so it is not the same operation as the engine's width helpers. The prompt's "wayland may consume it" is superseded by the tier rule (AGENTS.md: trust the architecture boundary). This is a justified deferral, not a skipped migration.

Test-only strip helpers left in place

Three strip_ansi copies remain under #[cfg(test)] (in tui.rs, terminal_surface.rs, and a chars().count() assertion in modal.rs). They are assertion utilities for rendered-output tests, not production parsers. Routing them through the engine risks the carefully-tuned terminal-surface tests for no production benefit, so they were left; the production duplicate parsers are gone.

Review-round fixes (wave 1)

  • O(n^2) tokenizer on malformed escapes (high): csi_len/string_control_len now return Some(rest.len()) for an unterminated sequence instead of None, so a dangling introducer is consumed as one zero-width token and the ansi_aware tokenizer stays linear on adversarial input. The production strip path (transform) was already single-pass linear; this hardens the deferred subsystem before it is wired. Test: tokenize_is_linear_on_unterminated_escapes.
  • slice_by_column style leak (medium): it now tracks an AnsiState and appends close_codes() before returning, so a slice ending mid-style does not leak SGR/OSC 8 into the next rendered content. Test: slice_by_column_closes_style_and_does_not_leak.
  • C1 (8-bit) consistency (medium): the tokenizer recognized 8-bit C1 CSI/OSC, but the tracker did not. AnsiState::process now also accepts the C1 CSI introducer and parse_osc8 the C1 OSC introducer, so C1-encoded styles/links carry across wrapped rows.
  • truncate_ansi ellipsis wider than field (low): guarded so the result never exceeds max_width (clips the ellipsis itself).
  • Dedup/cleanup: cluster_advance is exported from the engine and reused by wrap.rs (was duplicated); removed a dead binding in ansi_escape_len.
  • Added coverage: RGB/256-color SGR carry across wrap, OSC 8 preservation through truncate_ansi, and a CJK frame-helper test in ui/text.rs.

Review-round fixes (wave 2)

  • C1 ST terminator (U+009C): consume_string_control (production strip path) and string_control_len/parse_osc8 (engine) now treat the 8-bit C1 String Terminator as a valid terminator, not just BEL / 7-bit ST. Previously a C1-ST-terminated OSC/DCS would swallow the following visible text. Test added in strip_ansi_handles_osc_with_st_and_bel.
  • visible_width non-allocating fast path: when the input has no ESC and no 0xC2 lead byte (so no escape/C1 sequence is possible) it now sums widths directly instead of allocating a stripped copy. Control chars contribute 0 to UnicodeWidthStr either way, so this is identical output with no allocation on the common plain-text path.
  • slice_by_column overflow guard: start + len is now saturating_add.
  • SGR 21: kept as bold-off to match the pi-mono reference tracker, though ECMA-48 defines 21 as double-underline; this is the deferred subsystem and matches prior art. Colon (T.416) SGR sub-parameter separators are likewise not parsed (semicolon only) — noted, out of scope.

Deliberately NOT migrated (reviewer findings judged out of scope / not bugs)

tool_display.rs::fold (MAX_DISPLAY_CHARS) and truncate_inline use chars().count()/take(): these are char-count content budgets (how much text to show), not display-column layout, so char/grapheme count is the correct semantic — the same reasoning as wayland::truncate_chars. Converting them to display-width would wrongly halve a CJK-heavy preview. markdown.rs's continuation-prefix padding uses chars().count() on markers that are ASCII in practice. screen.rs::take_last_display already measures via the engine's display_width (it is a reverse-truncate algorithm, not a duplicate width impl). wrap.rs::styled_physical_row keeps per-char cells (documented above). None of these are production width/ANSI duplicates; leaving them is the smallest-correct scope.

Performance notes (no production regression)

Reviewers flagged allocations in the ansi_aware subsystem (tokenize returning a Vec, split(';').collect() in AnsiState::process, per-row active_codes/format!, the double tokenize in wrap_ansi). All are in the deferred, not-yet-wired subsystem and fine for real SGR input; left for when that path is wired to a real feature. The production transcript paths (display_width raw, wrap_to_width fast-path, push_wrapped_line per-cluster append) match the prior cost profile — no regression.

Known minor divergence: TAB width

The old ui/text.rs::visible_width counted a literal TAB as 1 (char count); the engine's visible_width reports 0 for control chars (TAB included), matching UnicodeWidthStr. For tab-indented frame/diff content the right-border padding shifts slightly. Both behaviors are wrong without true tab-stop expansion (the terminal renders TAB as a variable-width stop); tab expansion was left out of scope. Noted so it reads as a deliberate, bounded delta, not an oversight.

Reviewer-facing regression risks

  • Terminal wrap drift: the main risk is any divergence in wrap/truncate output for existing content. Mitigated by keeping display_width raw, preserving each migrated function's algorithm, and grapheme==char width equality for ASCII/CJK/single-emoji/combining (verified). Only ZWJ/flag/multi-RI sequences and the documented ui/text.rs CJK frame-overflow fix change output, all toward correctness.
  • let-chains: three collapsible_if sites in the engine use if A && let … let-chains (already used elsewhere in the tree); the tokenizer escape-introducer OR-group is bound to a local first to keep &&/|| precedence correct.
